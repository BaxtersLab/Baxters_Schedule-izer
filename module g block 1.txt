PURPOSE
--------------------------------------------------------------------------------
This block creates MainActivity (the BottomNavigationView host) and the
Navigation Component nav_graph.xml that wires all 5 tabs and their edit screens
together. After this block, the app shell is navigable — all 5 tab fragments
exist as stubs and the navigation graph is valid. The bottom nav bar is visible
and functional. Edit activities are registered in the graph.

This block also requests runtime permissions (RECORD_AUDIO, CAMERA,
POST_NOTIFICATIONS) from MainActivity so they are handled before any tab
needs them.

FILE: app/src/main/java/com/baxter/schedulaizer/ui/MainActivity.kt
Role: Single-activity host. Manages BottomNavigationView and NavController.
      Starts SchedulaizerService. Requests runtime permissions.

  Class: MainActivity : AppCompatActivity()

  Private properties:
    binding: ActivityMainBinding  (ViewBinding — generated from activity_main.xml)
    navController: NavController  (set in onCreate)

  onCreate(savedInstanceState):
    binding = ActivityMainBinding.inflate(layoutInflater)
    setContentView(binding.root)
    val navHostFragment = supportFragmentManager.findFragmentById(
        R.id.nav_host_fragment) as NavHostFragment
    navController = navHostFragment.navController
    binding.bottomNav.setupWithNavController(navController)
    startSchedulaizerService()
    requestRuntimePermissions()

  onSupportNavigateUp(): Boolean
    return navController.navigateUp() || super.onSupportNavigateUp()

  Private: startSchedulaizerService()
    val intent = Intent(this, SchedulaizerService::class.java)
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O)
        startForegroundService(intent)
    else
        startService(intent)

  Private: requestRuntimePermissions()
    Permissions to request (in one batch):
      Manifest.permission.CAMERA
      Manifest.permission.RECORD_AUDIO
      (API 33+) Manifest.permission.POST_NOTIFICATIONS
    Uses ActivityResultContracts.RequestMultiplePermissions.
    On result: if any permission is denied, show a Snackbar explaining
      what functionality is reduced (camera = no receipt capture,
      microphone = no voice agent, notifications = no alert pop-ups).
    Does NOT block the user from using the app if permissions are denied —
    features that need the permission simply show an in-context message
    when tapped.

--------------------------------------------------------------------------------
DESIGN DECISIONS
--------------------------------------------------------------------------------
- Single Activity architecture with Navigation Component. All 5 tabs are
  Fragments inside one Activity. EventEditActivity and BillEditActivity are
  separate Activities (not Fragments) because they are full-screen forms
  launched with arguments and benefit from independent back stack behavior.
- Permission requests are batched in MainActivity.onCreate() rather than
  lazily when a feature is first used. Reason: fewer interruptions mid-task.
  The user sees all permission dialogs on first launch.
- BottomNavigationView from Material Components — provides label, icon, and
  badge count support for future use (e.g., pending upload count badge).
